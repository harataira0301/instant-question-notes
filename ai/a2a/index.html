AI技術におけるA2Aとは何か

異なるAIエージェント同士が、仕事を依頼し合うための共通プロトコル。

Created: 2026-06-07

Overview

AI技術でいうA2Aは、多くの場合Agent2Agent Protocolを指します。A2Aは、異なる会社やフレームワークで作られたAIエージェント同士が、互いを見つけ、能力を理解し、タスクを依頼し、結果を受け取るためのオープンな通信ルールです。

たとえば、予定調整が得意なエージェント、メール作成が得意なエージェント、社内データ検索が得意なエージェントが別々に存在していても、A2Aに対応していれば、共通の形式でやり取りしやすくなります。

動作イメージ

なぜ必要か

AIエージェントは、単体で完結するよりも、別の専門エージェントや別システムと協力したい場面が増えています。しかし、それぞれが独自のAPIやデータ形式だけを持っていると、組み合わせるたびに個別の接続開発が必要になります。

A2Aは、その接続部分を共通化するための考え方です。相手がどのモデルを使っているか、内部でどんなツールを呼んでいるかを知らなくても、「どんな仕事ができるか」「どう依頼するか」「結果をどう受け取るか」をそろえます。

基本要素

要素 役割 初心者向けの理解
Agent Card エージェントの名前、説明、能力、エンドポイント、認証方式などを示す情報。 相手エージェントの名刺。
Task 依頼する仕事の単位。進行中、完了、失敗などの状態を持つ。 「この仕事をお願いします」という依頼票。
Message ユーザーやエージェントが送る入力、質問、補足など。 仕事のやり取りで交わす会話。
Artifact タスクの結果として返る成果物。テキスト、ファイル、構造化データなどを扱える。 完成したレポート、予定案、検索結果のような納品物。

MCPとの違い

A2Aと一緒に語られやすいものにMCPがあります。ざっくり言うと、MCPは「AIアプリやエージェントがツールやデータソースへつながるための規格」、A2Aは「エージェント同士が協力するための規格」です。

MCP: Agent -> Tool / Data
A2A: Agent -> Agent

たとえば、予定調整エージェントがカレンダーを読む時はMCP的な接続が役に立ちます。一方、その予定調整エージェントが別の出張手配エージェントへ仕事を依頼する時はA2A的な接続が役に立ちます。

使いどころ

  • 複数のAIエージェントを組み合わせて業務フローを作る。
  • ベンダーや部署ごとに違うエージェントを、共通の形式で連携させる。
  • 専門エージェントへタスクを委任し、進捗や成果物を受け取る。
  • 内部実装を隠したまま、外部から依頼可能なエージェントとして公開する。

注意点

A2Aは「エージェント同士を安全に何でも任せられる魔法」ではありません。実運用では、認証、認可、監査ログ、データの扱い、失敗時の回復、ユーザー確認が必要な操作の制御などを設計する必要があります。

また、A2Aに対応していても、エージェントの品質、権限、利用できるデータ、応答の正確性は別問題です。プロトコルは会話の形式をそろえるものなので、業務で使うには運用ルールと評価も欠かせません。

背景

A2AはGoogleが2025年4月に発表したプロトコルです。その後、2025年6月にGoogleはA2Aの仕様、SDK、開発者向けツールをLinux Foundationへ寄付し、より中立的なオープンプロジェクトとして進める流れになりました。

2026年時点では、AIエージェント間の相互運用性を高めるための代表的な取り組みのひとつとして扱われています。ただし、AIエージェント領域は変化が速いため、実装時は最新の仕様とSDKを確認するのが安全です。

Summary

AI技術におけるA2Aは、Agent2Agent Protocolの略として使われることが多く、AIエージェント同士が共通の形式で発見、依頼、進捗共有、成果物交換を行うための仕組みです。MCPが主にツールやデータへの接続を扱うのに対し、A2Aはエージェント同士の協力を扱います。便利な一方で、認証、権限、監査、品質評価まで含めて設計する必要があります。
